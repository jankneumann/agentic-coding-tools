- **`gitleaks` allowlist semantics: `stopwords` did not work in 8.24.3 for this case, `commits` did.** The first allowlist attempt used `stopwords = ["s3cr3t-32-byte-signing-key-abcdef"]` reasoning that gitleaks compares stopwords against the secret value. Possible explanation: the `[[allowlists]]` plural form may interact differently with `[extend] useDefault = true` than I assumed, or stopwords are only checked against rule-rendered matches not redacted ones. Worth filing a more reproducible test.

Lesson: wait first, refresh only if `mergeStateStatus` stays UNKNOWN past ~90s.

- **The `git commit --reset-author` trick saved a push.** Inline-overriding `git commit -c user.email=jankneumann@gmail.com` clobbered the noreply default and triggered GitHub's email privacy protection (`push declined`). `git commit --amend --reset-author --no-edit` regenerated authorship from the local config (the noreply form) without forcing me to specify an explicit email.
- **Rebase-merge refuses on PRs containing merge commits.** When `gh pr merge --rebase` fails with "This branch can't be rebased," the issue is structural (history includes merges), not transient. The fallback is squash (loses granular commits) or merge-commit (preserves them). For this PR, the operator chose merge-commit because the iteration history (codex review fixes, 4 secret-scan attempts, the cross-main conflict resolution) is worth keeping in the audit trail.
- **`git checkout origin/main` (detached HEAD) is the non-destructive cleanup-worktree reset.** When the user denies `git reset --hard origin/main` (correctly — it's destructive), detaching to `origin/main` achieves the same effect without leaving stale refs. Then `git push HEAD:main` lands the new commit as a fast-forward.
