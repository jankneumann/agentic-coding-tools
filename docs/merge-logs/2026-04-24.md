### Observations
- **Pre-existing main failures masquerade as stale-merge-commit failures.** Step 5b of the skill distinguishes transient / PR-specific / stale-base, but the scenario I hit (12 CVEs on main itself causing identical dep-audit failures on every PR) isn't covered. Differentiator: check main HEAD's own CI status, not just commit date. Worth adding to the skill's diagnostic flowchart.
- **Canonical-vs-runtime drift in `.claude/skills/` and `.agents/skills/`** is a recurring hygiene problem. Each `install.sh` run I did surfaced modifications to `vendor_review.py`, `autopilot.py`, `worktree/merge_worktrees.py` etc. that had been edited directly at runtime locations — violating the `feedback_canonical_skills.md` memory. Cumulative drift will need a reset commit at some point; I worked around it this session by `git restore`-ing each unrelated drift before committing my targeted changes.
- **Gitleaks scans diffs, not HEAD.** Inline `# gitleaks:allow` comments only silence findings on lines that CONTAIN the comment in the diff addition that introduced them. For a bugfix commit that adds the comments to lines introduced in an earlier commit, gitleaks still flags the original addition. Project-level `.gitleaks.toml` allowlist is the right tool when you need to silence historical findings without history rewrite.
- **Two P1s on #120 shared a failure mode (silent drop instead of explicit fail).** This is a transformation-pipeline anti-pattern worth watching for in similar curation/merge code — when two logical paths converge on the same output ID, silently preferring one violates the caller's contract if they asked for both.
- **Follow-up: #127** filed for `install.sh` to bundle `skills/shared/` alongside skill dirs. Without it, `worktree.py`'s isolation-detection feature silently falls back to the no-op stub at installed runtime — the whole cloud-harness scenario the feature was designed for.
