# Proposal: Phase-Record Compaction

**Change ID**: phase-record-compaction
**Status**: Draft
**Created**: 2026-04-25
**Author**: Claude (Opus 4.7)

## Summary

Unify the per-phase `session-log.md` entry and the coordinator `handoff_documents` row behind a single `PhaseRecord` data model that renders into both formats from one in-memory object, then wire autopilot phase boundaries to (a) write structured handoffs and (b) run heavy phases as ephemeral sub-agents that receive `(artifacts manifest + incoming PhaseRecord)` and discard their conversation on return. The combination shrinks Opus 4.7's runaway context accumulation in long autopilot runs while preserving full-fidelity artifacts across phase boundaries.

## Why

With Opus 4.7, the autopilot skill's single long-running conversation accumulates context faster than prior models can sustain. Artifacts (`proposal.md`, `tasks.md`, review-findings JSON files, validation reports) already capture phase outputs at full fidelity; the conversation scaffolding around them — file reads, trial edits, reasoning chains — is what fills the window without contributing to the durable record.

Two existing persistence mechanisms describe the same phase-boundary event:
- **`session-log.md`** — markdown phase entries for humans + git archaeology + the per-capability decision index (`docs/decisions/<capability>.md` regenerated by `make decisions`)
- **`handoff_documents`** — JSONB rows in the coordinator for programmatic retrieval by the next phase

The two formats overlap ~70% but drift independently because they are written by separate code paths. Right now `LoopState.handoff_ids: list[str]` exists at `skills/autopilot/scripts/autopilot.py:64` but is **never populated** — the structured handoff infrastructure is in place and unused. The 9 phase-transition boundaries already exist in `_HANDOFF_BOUNDARIES` (line 689-699); `_maybe_handoff` is called at each but receives only a transition-description string.

Unifying both formats behind one `PhaseRecord` removes drift by construction. Wiring autopilot to consume the structured handoff while running heavy phases as ephemeral sub-agents implements the user's mental model: at each phase boundary, equivalent to an interactive `/clear` that re-primes the next phase with **artifacts + a compact handoff**, never inheriting the prior conversation.

## What Changes

### 1. Unified `PhaseRecord` data model

New module `skills/session-log/scripts/phase_record.py`:

- `@dataclass PhaseRecord` with fields: `change_id`, `phase_name`, `agent_type`, `session_id`, `summary`, `decisions: list[Decision]`, `alternatives: list[Alternative]`, `trade_offs: list[TradeOff]`, `open_questions: list[str]`, `completed_work: list[str]`, `in_progress: list[str]`, `next_steps: list[str]`, `relevant_files: list[FileRef]`
- `Decision` sub-dataclass carries `title`, `rationale`, optional `capability` (kebab-case tag), optional `supersedes` (`<change-id>#D<n>` reference)
- `render_markdown()` — reproduces the existing phase-entry template; preserves backtick-delimited inline `` `architectural: <capability>` `` and `` `supersedes: ...` `` spans so the existing `make decisions` index regenerator continues working unchanged
- `to_handoff_payload()` — shape matches `HandoffService.write(...)` arguments
- `write_both()` — best-effort 3-step flow: append markdown → sanitize in-place → coordinator write_handoff. Each step independent; failures logged but not raised (matches existing sanitizer's exit-1-logs-and-continues behavior). Coordinator unavailability triggers fallback write to `openspec/changes/<id>/handoffs/<phase>-<N>.json`

### 2. Extend session-log template in place

Modify `openspec/schemas/feature-workflow/templates/session-log.md`:
- Add `## Completed Work` section (structured bullet list)
- Add `## Relevant Files` section (path + short description per entry)
- No template versioning — older changes simply lack these sections; renderers produce them when populated, omit when empty

Update `skills/session-log/SKILL.md` phase-entry template to match.

### 3. Retrofit all six phase-boundary skills in one pass

Replace `append_phase_entry(...)` calls with `PhaseRecord(...).write_both()` in:

| Skill | Current line | Phase name |
|---|---|---|
| `plan-feature` | SKILL.md:551 | `Plan` |
| `iterate-on-plan` | SKILL.md:283 | `Plan Iteration <N>` |
| `implement-feature` | SKILL.md:393 | `Implementation` |
| `iterate-on-implementation` | (similar pattern) | `Implementation Iteration <N>` |
| `validate-feature` | SKILL.md:727 | `Validation` |
| `cleanup-feature` | SKILL.md:328 | `Cleanup` |

`append_phase_entry()` remains in `extract_session_log.py` as a deprecated compatibility shim (other callers — e.g., `merge-pull-requests` writing to `merge-log` — keep working). Removal scheduled for a follow-up change after one release.

### 4. Autopilot Layer 1 — populate structured handoffs at phase boundaries

Modify `skills/autopilot/scripts/autopilot.py`:
- New helper `skills/autopilot/scripts/handoff_builder.py` exposing `build_phase_record(state, prev_phase, next_phase) -> PhaseRecord`. Per-phase builder functions extract relevant `LoopState` fields into structured `PhaseRecord` content
- Replace `_maybe_handoff` (line 712) description-string call with `handoff_fn(state, build_phase_record(state, prev, next))`
- Begin populating the existing-but-dead `LoopState.handoff_ids: list[str]` field (line 64). After each handoff write, append the returned `handoff_id`
- Add `LoopState.last_handoff_id: str | None` for fast lookup. Backward-compatible with existing autopilot snapshots (new fields default safely)
- Existing 9 boundaries in `_HANDOFF_BOUNDARIES` stay unchanged — this enriches payload, not boundaries

### 5. Autopilot Layer 2 — sub-agent phase isolation for heavy phases

New helper `skills/autopilot/scripts/phase_agent.py` wrapping `Agent(...)` invocation with a standard prompt scaffold: `(artifacts manifest, incoming PhaseRecord, phase task)`. Return contract: `(outcome: str, handoff_id: str)`.

Apply to the three heaviest phases:
- `IMPLEMENT` — runs in worktree isolation (`isolation: "worktree"`)
- `IMPL_REVIEW` — runs in shared checkout (read-only-ish)
- `VALIDATE` — runs in shared checkout

Lighter phases (`INIT`, `PLAN`, `PLAN_ITERATE`, `PLAN_REVIEW`, `IMPL_ITERATE`, `VAL_REVIEW`, `SUBMIT_PR`) stay in the driver conversation. Crash recovery: driver retries from scratch up to 3 times before escalating to operator (acceptable because phase artifacts are written incrementally — a re-run sees prior partial state).

### 6. Token-counting instrumentation

Add lightweight context-size measurement at autopilot phase boundaries (`anthropic.messages.count_tokens` or transcript-length proxy when offline). Emit telemetry via the coordinator's audit trail — measurable before/after comparison for the 30% peak-context-reduction success criterion. ~50 LOC.

### Out of scope

- Removing `append_phase_entry()` entirely (kept as shim, removed in follow-up)
- Changing the coordinator `handoff_documents` schema (current JSONB arrays accommodate `PhaseRecord` fields without migration)
- Touching non-workflow callers of session-log (e.g., `merge-pull-requests` continues to use `append_merge_entry` — separate unification opportunity later)
- Modifying `make decisions` per-capability index regenerator (markdown rendering preserves existing inline spans)
- Coordinating with `harness-engineering-features` proposal beyond using coordinator file locks at implement time (per discovery decision: proceed independently)

## Approaches Considered

### Approach A: Single coordinated change, staged work-packages (Recommended)

**Description**: One OpenSpec change shipping everything atomically, decomposed into 5 work-packages forming a DAG that the coordinator dispatches to parallel agents:

```
wp-contracts (schema)
    ↓
wp-phase-record-model (data model + tests)
    ↓
    ├─ wp-skills-retrofit (6 SKILL.md updates, parallel sub-tasks)
    └─ wp-autopilot-layer-1 (handoff wiring + instrumentation)
            ↓
        wp-autopilot-layer-2 (sub-agent isolation)
            ↓
    wp-integration (full test pass + smoke run)
```

**Pros**:
- Atomic merge — single review surface, contract clarity, no drift window
- Maximum parallelism within the change via coordinator work-queue
- Six skill retrofits + Layer 1 + instrumentation can run concurrently after model lands
- Maps directly onto user decision: "retrofit all phase-boundary skills in one pass"
- Single integration test pass validates end-to-end behavior before merge

**Cons**:
- Larger PR diff (~600-1000 LOC across many files)
- Reviewer needs to hold the full picture; harder to bisect a regression to a single concern
- Layer 2 (sub-agent isolation) is the riskiest piece — bundling it with Layer 1 means a Layer 2 regression blocks the simpler-to-validate handoff-format unification

**Effort**: L

### Approach B: Two coordinated changes — model+retrofit, then autopilot wiring

**Description**: Split into two sequential OpenSpec changes:

- **Change 1 — `add-phase-record-model`**: PhaseRecord + template extension + sanitizer integration + retrofit all six phase-boundary skills. Existing `append_phase_entry()` stays as shim. Tests verify markdown ↔ handoff round-trip and end-to-end skill behavior. Coordinator handoffs start being written, but autopilot doesn't yet consume them.
- **Change 2 — `autopilot-phase-compaction`**: Depends on Change 1. Layer 1 handoff wiring + Layer 2 sub-agent isolation + token-counting instrumentation. By this point PhaseRecord has live test coverage in production from Change 1 — autopilot consumes a known-good substrate.

Both changes still keep all 6 skill retrofits inside Change 1 (honors user decision 3 — "retrofit all in one pass" within their respective change). The split is between data-model + skill-API (Change 1) and autopilot orchestration (Change 2).

**Pros**:
- Smaller, faster-to-review PRs
- PhaseRecord ships and gets exercised by real workflow runs before the higher-risk autopilot rewrite consumes it
- Layer 2 (the riskiest piece) is isolated to Change 2 — easier to bisect and roll back independently
- Reduces blast radius if autopilot Layer 2 hits an unforeseen issue
- Cleaner ownership: skill-workflow team reviews Change 1, autopilot/coordinator concerns dominate Change 2

**Cons**:
- Two-PR coordination overhead — Change 1 must land before Change 2 starts
- Brief window where structured handoffs are written but autopilot still uses description-strings — measurable but harmless drift
- Reviewer of Change 1 must trust the design.md description of Change 2's consumption pattern; cannot validate end-to-end compaction in Change 1 alone

**Effort**: M + M (instead of L)

### Selected Approach

**Approach A — Single coordinated change, staged work-packages** (confirmed at Gate 1, 2026-04-25, no modifications).

Rationale:
- The user's brief explicitly asks for atomic scope ("retrofit all phase-boundary skills in one pass") and the decision matrix from discovery favors a single coordinated push
- Coordinated tier work-packages give us the parallelism benefits Approach B provides via PR boundary, without splitting the change
- The integration test pass at `wp-integration` validates the full compaction story (PhaseRecord round-trip + autopilot Layer 1 + Layer 2 + token reduction measurement) before any merge — this is exactly what Approach B sacrifices
- Risk mitigation for Layer 2: the work-package DAG positions `wp-autopilot-layer-2` as the last leaf before integration. If it hits trouble, we can land everything else (Layers 0+1 = unified format + structured handoffs) and defer Layer 2 to a follow-up by trimming `wp-autopilot-layer-2` from the merge — effectively recovering Approach B's escape hatch when needed

The size penalty (~600-1000 LOC) is real but the change is mostly mechanical (6 SKILL.md retrofits all follow the same pattern) plus one new module (`phase_record.py`) and two thin wrappers (`handoff_builder.py`, `phase_agent.py`).

## Success Criteria

1. **Format unification verified**: All six phase-boundary skills produce identical content in `session-log.md` and `handoff_documents` for the same phase, verifiable by reading both back and asserting field equivalence after the test suite runs at least one full feature workflow
2. **Per-capability decision index unchanged**: `make decisions` regenerates `docs/decisions/<capability>.md` with byte-identical output (modulo timestamps) before vs. after the change — proving the `architectural:` and `supersedes:` inline-span round-trip is preserved
3. **Coordinator-unavailable graceful degradation**: With coordinator stopped, `PhaseRecord.write_both()` writes to `openspec/changes/<id>/handoffs/<phase>-<N>.json` and the workflow continues with a logged warning (no exception raised, no skill failure)
4. **Autopilot context reduction measured**: Token-counting instrumentation reports ≥30% peak-context-window reduction on a representative mid-size feature run (e.g., a feature with ≥3 work packages, ≥2 review iterations, ≥1 validation cycle), comparing pre-Layer-2 driver transcript size vs. post-Layer-2
5. **Sub-agent isolation enforced**: After `IMPLEMENT`, `IMPL_REVIEW`, or `VALIDATE` phase returns, the driver `LoopState` and conversation contain only `(outcome, handoff_id)` from the sub-agent — never the sub-agent's transcript content. Verifiable via test that asserts driver context delta is bounded by handoff payload size
6. **Crash recovery works**: Simulated sub-agent crash mid-phase triggers driver retry up to 3 times with the same incoming handoff; final retry failure raises `PhaseEscalationError` with the failing phase name and last-attempt error context
7. **All existing tests pass**: ~1690 lines of existing tests across session-log, handoffs, and autopilot continue to pass without modification (only additions allowed). New tests cover PhaseRecord round-trip, sanitizer integration in `write_both`, fallback to local-file handoff, autopilot Layer 1+2 dispatch, token instrumentation
8. **Backward compatibility for snapshots**: Existing `LoopState` JSON snapshots (from in-flight autopilot runs at deploy time) load successfully into the updated dataclass without manual migration — new fields (`last_handoff_id`) default to `None`; existing `handoff_ids` field begins being populated
