1. **Package lives at packages/gen-eval/ (new top-level convention)** `architectural: gen-eval-framework` — User clarified that agentic-coding-tools is conceptually a coordinator service + shared component libraries, with multiple future extractions planned. Introducing the packages/ convention with one example is cheaper than retrofitting later with three. 2. **Framework-only data split; descriptors stay in consumers** `architectural: gen-eval-framework` — Matches the stated architectural intent 'code in agentic-coding-tools, configuration in consumer repos'. Package ships .py + schemas/ + dtu/ templates + tests/fixtures/ + examples/. agent-coordinator keeps its descriptor.yaml + manifests/ + scenarios/ as project test fixtures.
3. **Optional [mcp] extra inside the package** `architectural: gen-eval-framework` — Two consumer profiles in the ecosystem: agent-coordinator wants MCP routing (it's a service), agentic-assistant wants pure Python (skills do orchestration). Optional extra is the only design that serves both without forcing a wrong dependency on either. mcp_service.py and clients/mcp_client.py wrap their fastmcp imports in try/except per design D4.
4. **Move evaluation/metrics.py into the package** `architectural: gen-eval-framework` — Resolves the only reverse coupling. GenEvalMetrics is named for gen-eval, only consumed by gen-eval's reports.py, and has no other consumers. Alternatives (Protocol injection, no-op stub, callback shim) all add indirection for zero gain.
5. **Full cutover in one atomic change** — Phased approach saves no work (the follow-up has the same skill/Dockerfile updates) and creates a drift window with two sources of truth. The convention isn't really set until the coordinator actually uses the package.
6. **agentic-assistant adoption proven via examples doc only, not cross-repo PR** — User chose the compromise option. Real cross-repo PR costs coordination overhead (separate CI, separate review cycle). The quickstart doc is verifiable inside this repo; if it turns out to be wrong, the agentic-assistant adoption will surface it.

### Alternatives Considered
- Strangler: gen_eval re-exports from agent_coordinator initially: rejected because Would require gen-eval to depend on agent-coordinator, inverting the dependency graph. The package should not depend on the service that consumes it.
- Two-package split (gen-eval-core + gen-eval-mcp): rejected because Adds a second pyproject + release surface + test job for marginal benefit. Optional [mcp] extra gives the same install-time flexibility with one package.
- Use evaluation/gen_eval/ at repo root as package home: rejected because Namespace ambiguity during cutover (root vs. agent-coordinator/ at different depths); single-purpose dir name forces a different convention for the next extraction anyway.

### Trade-offs
- Accepted Single atomic change (~30 file moves + Dockerfile + skills + pyproject) over Phased two-change rollout because No drift window; the convention is set the moment the change lands.
- Accepted Path-only install via uv add over PyPI publishing in this change because PyPI is real release infrastructure (CI publish workflow, version-bump policy, namespace claim, trusted publisher setup). Worth a dedicated change after the first external consumer proves the path-install works.
- Accepted Examples doc as agentic-assistant adoption proof over Cross-repo PR to agentic-assistant because Lower coordination cost; doc can be verified inside this repo.
