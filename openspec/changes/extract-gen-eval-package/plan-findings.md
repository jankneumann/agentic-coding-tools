### Findings

| # | Type | Criticality | Description | Proposed Fix |
|---|------|-------------|-------------|--------------|
| F1 | feasibility | **critical** | **`metrics.py` move is wrong-grained.** Design D3 says "Move `agent-coordinator/evaluation/metrics.py` into the package as `gen_eval/metrics.py`." But `evaluation/metrics.py` defines 11 classes + 1 function; only `GenEvalMetrics` (1 class) is gen-eval-specific. The other 10 classes (`TimingMetric`, `TokenUsage`, `CorrectnessMetrics`, `CoordinationMetrics`, `SafetyMetrics`, `ParallelizationMetrics`, `TaskMetrics`, `AggregatedMetrics`, `TrialMetrics`, `MetricsCollector`) and `compute_effect_size` are coordinator-domain and are consumed by `agent-coordinator/evaluation/ablation.py`, `evaluation/reports/generator.py`, and four test files. Moving the whole file would either (a) break those consumers or (b) force the coordinator to circularly import its own classes back from the gen-eval package. | D3 (design.md), task 2.4 (tasks.md), and the spec's MODIFIED Module discovery requirement must specify **extracting only `GenEvalMetrics`** (and `compute_effect_size` only if it is genuinely needed by gen-eval's reports.py — check) into `gen_eval/metrics.py`. The rest of `evaluation/metrics.py` stays where it is. wp-framework-move's verification and scope need to reflect this surgical move, not the whole-file move. |
| F2 | completeness | **high** | **Coordinator test suite import sites are not in the plan.** Task 4.3 covers `agent-coordinator/src/` only. But `agent-coordinator/tests/test_evaluation/test_gen_eval/` has 29 test files all importing `from evaluation.gen_eval.*`, and `agent-coordinator/tests/test_check_docker_imports.py` embeds literal `from evaluation.gen_eval import mcp_service` strings as test fixtures. Two cases: (a) tests that are moving into the package, which need import rewrites + relocation; (b) tests that stay (`test_check_docker_imports.py`), which need their literal string fixtures updated. | Update task 3.1 to name the correct path (`agent-coordinator/tests/test_evaluation/test_gen_eval/`, not `tests/evaluation/gen_eval/`). Add a sub-task to task 4.3 or a new task that explicitly updates the leftover string fixtures in `test_check_docker_imports.py`. Update wp-package-tests's `scope.write_allow` to include `agent-coordinator/tests/test_evaluation/test_gen_eval/**` (the source it must delete after moving). Or move the test relocation into wp-framework-move so it's atomic. |
| F3 | completeness | **high** | **Missed skill import sites.** D9 row 4 (`skills/gen-eval-scenario/SKILL.md`) says "no change" — but the SKILL.md at line 172 contains `from evaluation.gen_eval.models import Scenario` as part of a Python validation snippet. Separately, `skills/playwright-validator/scripts/cli.py:141` contains a runtime `from evaluation.gen_eval.openspec_seed import ...` import — the plan updates the SKILL.md but not the script. Plus `skills/playwright-validator/scripts/findings.py:112` references the legacy path in a docstring. | Add `skills/gen-eval-scenario/SKILL.md` to D9 and to task 5.x. Update task 5.3 to include both `SKILL.md` AND `scripts/cli.py` (and the `sys.path` manipulation — once gen-eval is a real installed package, the `sys.path.insert(0, .../agent-coordinator)` hack is unnecessary and should be removed or kept only as legacy fallback). Update docstring in `scripts/findings.py`. Update `wp-skills-update`'s `write_allow` to cover `skills/playwright-validator/scripts/**`. |
| F4 | feasibility | **high** | **Docker build context change breaks Railway deployment.** Design D8 hand-waves "Likely cleanest: change `docker build` context to the repo root rather than `agent-coordinator/`, with `-f agent-coordinator/Dockerfile`." But three deploy/CI surfaces consume this context, and the plan addresses only the Dockerfile + a README note: (a) `.github/workflows/ci.yml:178` sets `context: agent-coordinator` for the docker-build CI step. (b) `agent-coordinator/docker-compose.yml:205` sets `context: .`. (c) `agent-coordinator/railway.toml` configures Railway to build with `agent-coordinator/` as the service root and `dockerfilePath = "Dockerfile"` — Railway will not see `../packages/gen-eval/` if the service root stays at `agent-coordinator/`. | D8 must pick an explicit strategy and document the Railway implication. Options: **(a) Move build context to repo root** — requires updating ci.yml, docker-compose.yml, AND reconfiguring Railway service-root + dockerfilePath. **(b) Pre-build wheel and COPY it** — gen-eval is built to a wheel in CI/locally, the wheel file is committed or copied into `agent-coordinator/dist/` at build start, and the Dockerfile installs the wheel via `uv pip install dist/gen_eval-*.whl`. Keeps Railway's service root at `agent-coordinator/`. **(c) Use a docker bind-mount via .dockerignore + symlink trick** — fragile, do not recommend. Plan should pick one, document it in D8, and add the corresponding tasks/scope to wp-coordinator-migrate (including `.github/workflows/ci.yml`, `docker-compose.yml`, `railway.toml`). |
| F5 | parallelizability | medium | **Cross-package scope coupling for `.github/workflows/ci.yml`.** `wp-package-tests` owns `.github/workflows/ci.yml` (to add the new `gen-eval-tests` matrix job). If F4 is fixed via Option (a) [repo-root build context], `wp-coordinator-migrate` also needs to edit `.github/workflows/ci.yml` (to change the docker-build context line). Two parallel packages writing to the same file = merge conflict during integration. | Either (a) keep `.github/workflows/ci.yml` exclusively in `wp-coordinator-migrate` and have it ALSO add the gen-eval-tests job (combining concerns), (b) sequence wp-package-tests after wp-coordinator-migrate, or (c) pick F4 Option (b) which keeps the docker-build context at agent-coordinator and avoids the conflict entirely. Recommended: F4-option-(b) + this becomes a non-issue. |
| F6 | clarity | medium | **Task 3.1 source path is wrong.** Task 3.1 reads "Relocate gen-eval's existing unit tests from `agent-coordinator/tests/evaluation/gen_eval/`". Actual path is `agent-coordinator/tests/test_evaluation/test_gen_eval/` (29 files). The implementer will spend time hunting for a directory that doesn't exist. | Fix task 3.1 text. |
| F7 | completeness | medium | **D7 hedge "if anything actually exists there" is wrong.** Design D7 says "`evaluation/gen_eval/fixtures/**` (at repo root) → MOVED to `packages/gen-eval/tests/fixtures/` (if anything actually exists there)". The directory contains `sample-frontend/index.html` and `sample-frontend/specs/sample.spec.md` — content exists. The conditional phrasing might cause an implementer to skip them. | Remove the hedge. State concretely what moves. |
| F8 | testability | medium | **No verification that the published wheel's metrics surface is correct.** Once F1 is fixed (only `GenEvalMetrics` extracted), there's no test asserting that the package's metrics module exposes `GenEvalMetrics` AND does NOT expose the unrelated 10 classes. Without this, a future refactor could re-introduce the coupling and no one would notice. | Add a test under `packages/gen-eval/tests/` (or extend task 3.3) that asserts the public surface of `gen_eval.metrics` matches exactly the intended set. Could be as simple as `assert set(dir(gen_eval.metrics)) - dunder_set == {'GenEvalMetrics', 'compute_effect_size'}` (or whatever the final surface is). |
| F9 | clarity | medium | **wp-framework-move locks list contains a stale path.** `wp-framework-move.locks.files` includes both `packages/gen-eval/src/gen_eval` (correct) AND `agent-coordinator/evaluation/gen_eval` (legitimate — being deleted) AND `agent-coordinator/evaluation/metrics.py` (per F1, no longer fully owned by this WP — coordinator-domain code that remains there). | After fixing F1, refine the lock list: `agent-coordinator/evaluation/metrics.py` is *partially* edited (the `GenEvalMetrics` class is excised) but not deleted. Document this lock semantics — a partial-file lock for "edit GenEvalMetrics out" rather than "own metrics.py". |
| F10 | consistency | low | **Spec MODIFIED requirement still talks about whole-file move.** The MODIFIED Module discovery requirement says "The previously existing reverse coupling (`evaluation/metrics.py` importing into gen-eval's `reports.py`) SHALL be resolved by moving `metrics.py` into the package as `gen_eval/metrics.py`, since `GenEvalMetrics` is gen-eval-specific by name and has no other consumers." Last clause is factually wrong — `GenEvalMetrics` has no other consumers; the *file* it lives in has 10 other consumers. | Adjust the requirement language to "by moving `GenEvalMetrics` (and any helpers it directly depends on) into the package as `gen_eval/metrics.py`." |
| F11 | clarity | low | **Parallelizability assessment missing from plan.** The plan implicitly expects 4-way parallel after wp-framework-move, but task / work-package text doesn't summarize the dependency-graph shape or call out the converge point. | Add a 2-line "Parallelizability" note to tasks.md after section 6: `Independent (post-2.C): wp-package-tests, wp-coordinator-migrate, wp-skills-update, wp-examples-doc. Max parallel width: 4. Converge: wp-integration.` |
| F12 | assumptions | low | **Implicit assumption: editable install survives Docker multi-stage build.** D5 specifies `editable = true` for the path dep, which is great for dev. But the multi-stage Dockerfile copies the venv from the builder stage (line `COPY --from=builder /app/.venv /app/.venv`). Editable installs include an `.pth` file pointing to the source on disk — if the source location in the runtime stage differs from the build stage path, the import will fail at runtime. | Document in D5 that the editable install works in the multi-stage build because the source is COPIED into the same path in both stages, OR switch to `editable = false` for production builds (uv supports per-environment overrides). |
