### Decisions
1. **Subprocess fallback over native checkpoint-then-synthesize** `architectural: multi-vendor-review` — Reuses the existing consensus_synthesizer.py CLI binary verbatim instead of forking synthesis logic. Future synthesizer fixes inherit automatically; no two-path divergence risk.
2. **Reuse .review-cache/ layout used by review_dispatcher.py CLI** `architectural: multi-vendor-review` — Same directory, filenames, and manifest format as the CLI dispatcher. Eliminates path translation and a second loader; the fallback subprocess reads checkpoint files without any conversion.
3. **Observable recovery via two ConvergenceResult fields** `architectural: multi-vendor-review` — recovered_via_fallback: bool + fallback_diagnostics: dict | None directly on the dataclass. Side-channels (env var, thread-local, audit-only) hide recovery from callers that legitimately need to know.
4. **Re-raise original synthesis exception when fallback also fails** `architectural: multi-vendor-review` — Subprocess stderr tail attaches as forensic data (Python 3.11+ __notes__) but the surface exception is the original synthesis error, since that is the diagnostically interesting one.
5. **Audit emission failures do not mask recovery** `architectural: multi-vendor-review` — Coordinator unavailability when emitting convergence.fallback_recovered must not un-recover a successful recovery. Wrap in try/except + logger.warning. Mirrors coordination-bridge convention.

### Alternatives Considered
- Approach 2 — Rebuild converge() to checkpoint-then-synthesize natively (no subprocess): rejected because Larger refactor for the same end-state durability. The load-bearing change is the pre-synthesis checkpoint itself; whether recovery uses subprocess or native synthesis is secondary. Keeping the existing CLI as the recovery path reuses tested code instead of forking it.
- Approach 3 — Ship CLI fallback now, refactor to native checkpointing later: rejected because Two proposals where one suffices. Approach 1 already gives the durability + observability we need; pre-committing to a future refactor creates phantom roadmap that may never be needed.
- Bundle the consensus_synthesizer.py:59 line_range type bug fix into this proposal: rejected because User explicitly requested the bug be filed separately. The recovery mechanism in this proposal protects against that bug AND its analogues, so coupling them would mix systemic recovery with a specific bug fix.

### Trade-offs
- Accepted Subprocess overhead on the recovery path over Native re-implementation of synthesis in converge() because Recovery path is cold (only fires on synthesis failure). Subprocess overhead is negligible compared to the cost of dual-maintaining two synthesis implementations.
- Accepted Two error paths exist (in-process + subprocess) on the failure branch over Single architectural pattern via Approach 2 because Both paths invoke the SAME synthesizer source (consensus_synthesizer.py). The duality is subprocess-vs-in-process, not two synthesis implementations.
- Accepted Permissive line_range schema (oneOf dict|str|null) over Strict schema rejecting the buggy string shape because The contract documents the wire format AS IT IS, including the malformed shape that triggered the original incident. Tightening the schema here would break callers using the in-memory synthesizer before the recovery could fire. The bug fix tightens it in the separate follow-up proposal.
