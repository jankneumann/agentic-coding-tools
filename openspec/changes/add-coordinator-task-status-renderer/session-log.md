### Alternatives Considered
- Sidecar file tasks.status.md (Approach 2 in proposal): rejected because Rejected at Gate 1. Keeps tasks.md untouched but leaves the file the user actually opens still stale. Splits PR diffs across two files and asks every downstream consumer (cleanup-feature, etc.) to learn the sidecar's existence. Doesn't fix the stated problem.
- Auto-updating PR comment via GitHub webhooks (Approach 3 in proposal): rejected because Rejected at Gate 1. Requires a PR to exist (no in-flight visibility before PR open), introduces GH webhook ops surface, and doesn't help local-workflow users. Repo ethos favors git-tracked artifacts.
- Lazy seeding on first renderer invocation: rejected because Rejected during design. A renderer that silently mutates the coordinator on read is hard to reason about. Discovery Q2 explicitly chose Gate-2 seeding.
- Pre-commit hook only (no post-merge): rejected because Rejected at trigger-point follow-up. Covers PR-review case but leaves the local-pull workflow stale. Marginal cost of adding post-merge (~10 lines, fires only on tasks.md-touching merges) is small.
- Hard validation gating (refuse task advancement without evidence): rejected because Out of scope. Current problem is display drift, not fabricated completions. The result.evidence_uri convention is wired so a future change can add gating without schema work, but this proposal stays a projection-only solution.

### Trade-offs
- Accepted Per-commit churn in tasks.md history (5-15 line diffs as status advances) over A static tasks.md that stays stale until someone manually ticks boxes because Display drift is the explicit problem. A small, deterministic diff each commit is acceptable in exchange for tasks.md always matching coordinator state at commit time.
- Accepted Silent clobbering of hand-edits inside the managed block over A merge-rule or warning that preserves edits at the cost of state drift because The point of the managed block is to remove the human from the loop for status fields. Explicit <!-- GENERATED: begin/end --> markers make the ownership obvious; the alternative reintroduces the discipline failure mode.
- Accepted Coordinator HTTP latency added to every commit and pull that touches tasks.md over Background async reconciliation because Latency is bounded by the bridge's short timeout (~2s) and the path falls back cleanly to stale-marker on failure. Synchronous renders are easier to reason about and reviewable in commit diffs.

### Context
Planned a coordinator-owned managed block in tasks.md that renders task status (checkboxes, claimed_by, completion timestamps) from coordinator-recorded issues. The renderer is invoked by pre-commit and post-merge hooks, falls back to a stale marker on coordinator failure, and is seeded by /plan-feature Gate 2 approval. No new coordinator schema; reuses existing managed-block helpers from plan-roadmap and HTTP helpers from coordination-bridge.
