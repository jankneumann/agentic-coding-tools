# Design: Add Phase Session Logs

## Architecture

### Session Log as Living Artifact

```
Phase 1: /plan-feature          → appends "Plan" section
Phase 2: /iterate-on-plan       → appends "Plan Iteration" section
Phase 3: /implement-feature     → appends "Implementation" section
Phase 4: /iterate-on-impl       → appends "Implementation Iteration" section
Phase 5: /validate-feature      → appends "Validation" section
Phase 6: /cleanup-feature       → appends "Cleanup" section (replaces current Step 5b)
                                → archived with change
```

Each phase appends; no phase overwrites. The file grows incrementally across sessions and agents.

### Concurrency Model

In parallel execution, each agent works in its own worktree on a separate branch. Each agent appends to its own copy of `session-log.md`. During the integration merge step (`merge_worktrees.py`), session-log.md entries from different branches are combined. Since the file is append-only, git merge handles most cases automatically. For true conflicts (two agents appending at the same position), the integration merge resolves by chronological ordering based on the phase entry date in each header.

### Merge Log as Dated Entries

```
/merge-pull-requests → docs/merge-logs/2026-03-31.md
                     → docs/merge-logs/2026-04-01.md
                     → ...
```

Each merge session appends to the day's file. Multiple merge sessions on the same day append to the same file with separator headers.

## Session Log Format

### Phase Entry Template

Each phase appends a section following this structure:

```markdown
---

## Phase: <phase-name> (<YYYY-MM-DD>)

**Agent**: <agent-type> | **Session**: <session-id-if-available>

### Decisions
1. **<Decision title>** — <rationale>

### Alternatives Considered
- <Alternative>: rejected because <reason>

### Trade-offs
- Accepted <X> over <Y> because <reason>

### Open Questions
- [ ] <unresolved question>

### Context
<2-3 sentences of phase-specific context: what was the goal, what happened>
```

### Phase Names

| Skill | Phase Name |
|-------|-----------|
| plan-feature | `Plan` |
| iterate-on-plan | `Plan Iteration <N>` |
| implement-feature | `Implementation` |
| iterate-on-implementation | `Implementation Iteration <N>` |
| validate-feature | `Validation` |
| cleanup-feature | `Cleanup` |

### Iteration Numbering

For phase names with `<N>` (Plan Iteration, Implementation Iteration), the iteration number is auto-incremented:

1. Read `session-log.md` (if it exists)
2. Count existing `## Phase: Plan Iteration` (or `Implementation Iteration`) headers
3. N = count + 1

This is scoped to the change-id (the file itself). No external counter needed.

### File Initialization

When the first phase runs and no `session-log.md` exists yet, it creates the file with a header:

```markdown
# Session Log: <change-id>

> Decision rationale, trade-offs, and alternatives recorded at each workflow phase.
> Auto-generated by workflow skills. Sanitized before commit.
```

Then appends the first phase entry.

## Merge Log Format

### Daily Entry Structure

```markdown
# Merge Log: <YYYY-MM-DD>

---

## Session: <HH:MM> (<agent-type>)

### PRs Processed

| PR | Origin | Action | Rationale |
|----|--------|--------|-----------|
| #123 | openspec | merged | CI green, approved, no conflicts |
| #124 | codex | closed | Obsolete — fix already on main |
| #125 | dependabot | skipped | Pinned dependency, will upgrade next quarter |

### Vendor Review Findings
- PR #126: 2 confirmed findings (both addressed), 1 unconfirmed (accepted as low risk)

### User Decisions
- Skipped all Renovate PRs per user request (pinning React 18 until migration complete)

### Observations
- 3 Codex PRs touched the same auth middleware — may indicate a recurring issue
```

### File Lifecycle

- Created on first merge session of the day
- Appended to on subsequent sessions the same day
- Not archived or rotated — git history provides natural versioning
- `.gitkeep` in `docs/merge-logs/` to ensure directory exists

## Append-Sanitize-Verify Flow

Every skill follows the same 3-step pattern at its session-log step:

```
1. APPEND: Agent writes phase entry to session-log.md (or merge-log)
2. SANITIZE: Run sanitize_session_log.py on the file (in-place)
3. VERIFY: Agent reads sanitized output and checks:
   a. All phase entry sections are present (or intentionally omitted per "no decisions" rule)
   b. No [REDACTED:*] markers in narrative prose where original had no secrets
   c. Markdown structure is intact (headers, lists, tables parseable)
   - If over-redacted: rewrite the affected section without the triggering pattern, re-sanitize (one attempt max)
   - If clean: proceed to commit
   - If sanitization exits non-zero: do NOT commit session-log.md, log warning, continue workflow
```

### Sanitization Integration

```bash
# For session-log.md (in change directory)
python3 "<skill-base-dir>/../session-log/scripts/sanitize_session_log.py" \
  "openspec/changes/<change-id>/session-log.md" \
  "openspec/changes/<change-id>/session-log.md"

# For merge-log (in docs directory)
python3 "<skill-base-dir>/../session-log/scripts/sanitize_session_log.py" \
  "docs/merge-logs/<date>.md" \
  "docs/merge-logs/<date>.md"
```

Note: sanitize_session_log.py already supports in-place operation (same input/output path).

## extract_session_log.py Refactoring

### What to Keep

- `generate_self_summary_prompt()` — useful as a reference template, though skills will embed their own phase-specific prompts
- `sanitize_session_log.py` — unchanged, comprehensive tests pass

### What to Retire

- `try_extract_claude_transcript()` — Tier 1 JSONL parsing, fragile and Claude-specific
- `try_extract_from_handoffs()` — Tier 2 handoff compilation, redundant when agents write directly
- `_parse_jsonl_messages()` — supporting function for Tier 1
- `_format_transcript_as_session_log()` — supporting function for Tier 1
- `_format_handoffs_as_session_log()` — supporting function for Tier 2
- `main()` CLI entry point — the 3-tier orchestration is no longer needed

### What to Add

- `append_phase_entry()` — Python function that appends a phase entry to an existing session-log.md (creates if missing), handling the header and separator
- `append_merge_entry()` — Python function that appends a merge session entry to a dated merge-log file (creates if missing)
- Both functions are called by the SKILL.md instructions (agents can also write directly; the functions provide consistency)

## Integration Points

### Skills That Commit (append before final commit)

1. **plan-feature** — New Step 8.5 between "Commit, Push, Pin" and "Present for Approval"
   - Actually: include session-log.md in Step 8's `git add` so it's part of the planning commit
2. **implement-feature** — New Step 8.5 between "Evidence Collection" and "Push and Create PR"
   - Include session-log.md in the PR commit
3. **cleanup-feature** — Replace existing Step 5b with the new append pattern

### Skills That Don't Commit (append + commit as new step)

4. **iterate-on-plan** — New final step after "Present Summary" (Step 11.5)
5. **iterate-on-implementation** — New final step after "Present Summary" (Step 12.5)
6. **validate-feature** — New final step after "PR Comment" (Step 13.5)

### Merge Log (separate artifact)

7. **merge-pull-requests** — New final step after "Summary" (Step 12.5)
   - Writes to `docs/merge-logs/YYYY-MM-DD.md` instead of change directory

### Skills That Don't Get Session Logs

- **parallel-review-plan** — Dispatched to external agents, stateless
- **parallel-review-implementation** — Dispatched to external agents, stateless
- **explore-feature** — Exploratory, no change directory
- **bug-scrub / fix-scrub** — Diagnostic, separate artifact lifecycle
