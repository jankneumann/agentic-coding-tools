# Design: Software Factory Tooling

**Change ID**: `add-software-factory-tooling`

## Goals

- Make software-factory practices reusable by external projects, not just by this repository
- Add first-class support for public vs holdout scenarios
- Add a DTU-lite path that starts from public docs and becomes gate-worthy only after fidelity checks
- Turn validation failures into structured rework artifacts
- Mine archived OpenSpec changes into reusable exemplars and scenario seeds
- Dogfood the full flow on this repository

## Non-Goals

- Fully autonomous “no human in the loop” operation
- Perfect third-party system emulation
- A separate parallel workflow outside OpenSpec
- Replacing existing validation-report, change-context, or session-log artifacts

## Design Decisions

### D1: Scenario visibility is modeled in a manifest, not inferred from paths

**Decision**: Add a scenario-pack manifest that records visibility (`public`, `holdout`), provenance, determinism, ownership, and promotion status for each scenario or scenario group. Path layout may still separate packs on disk, but behavior is driven by manifest metadata.

**Rationale**: File naming conventions are too weak for workflow gating. Visibility must be queryable by gen-eval, validation, and archive mining without heuristic guessing.

**Rejected alternatives**:
- Infer visibility solely from directory names like `scenarios/public/` and `scenarios/holdout/`
- Store visibility only in scenario YAML frontmatter

### D2: Holdout enforcement uses directory structure, manifest metadata, and skill-level filtering

**Decision**: Scenarios are organized into separate directories (`public/` and `holdout/` under each scenario category). The scenario-pack manifest confirms visibility metadata for each entry. Skill logic enforces the boundary at runtime — implementation skills read only from `public/` directories, while validation and cleanup gates read both. No gitignore-style exclusion policies are needed; clear path conventions plus manifest confirmation are sufficient.

**Rationale**: Directory structure makes intent immediately visible to anyone browsing the repo. The manifest provides a machine-readable safety check if a tool accidentally globs across directories. Skill-level filtering is the runtime enforcement layer. Three reinforcing mechanisms without the complexity of filesystem-level access policies.

**Rejected alternative**: Manifest-only enforcement (no directory split) — makes holdout status invisible in the filesystem and relies entirely on tools reading metadata correctly.

### D3: DTUs start as “DTU-lite” scaffolds with fidelity reports; live probes are optional

**Decision**: DTU generation from public SDK/API docs produces a scaffold, not a trusted oracle. Every DTU gets a fidelity report describing:
- sources ingested
- unsupported areas
- probe results (when available)
- conformance score
- holdout eligibility

Live system probing is optional and enhances the fidelity score but is never required. DTUs bootstrapped from docs alone are eligible for public scenarios. Holdout eligibility requires either live probe confirmation or explicit operator approval.

**Rationale**: Public docs are useful for bootstrapping but insufficient for blind trust. Making probes optional keeps bootstrap fast and offline-capable while the fidelity report communicates the confidence level. Teams that need holdout-grade DTUs can add probes incrementally.

**Rejected alternatives**:
- Treat doc-derived twins as production-grade immediately
- Require live probes before any scenario use (blocks bootstrap on live system availability)

### D4: Validation emits a machine-readable rework report

**Decision**: `/validate-feature` produces `rework-report.json` with deterministic routing fields:
- failed scenario IDs
- visibility
- likely requirement refs
- likely package/file owners
- implicated interfaces
- recommended next action (`iterate`, `revise-spec`, `defer`, `block-cleanup`)

`/iterate-on-implementation` consumes that artifact rather than reparsing freeform validation prose.

**Rationale**: The workflow already has iteration skills, but they need a machine-readable control signal if validation is going to drive rework reliably.

### D5: Process analysis is a first-class optional artifact generated during validation

**Decision**: Add `process-analysis.md` and `process-analysis.json` as optional OpenSpec artifacts generated by `/validate-feature`. Cleanup and merge gates consume the artifact read-only but do not regenerate it. The artifact summarizes convergence behavior: loops taken, findings addressed, flaky scenarios, time to first pass, churn, and failure classes.

**Rationale**: `/validate-feature` is the phase with the most convergence data (scenario results, iteration counts, finding resolutions). Generating in a single location avoids split-generation complexity while ensuring the artifact is available during iteration loops when it's most useful. Archive mining benefits from explicit process outcomes rather than inference from commit history.

**Rejected alternative**: Generate incrementally in both validate and cleanup — richer but adds two generation points and merge complexity for the artifact.

### D6: Archive mining uses deterministic normalization before advanced retrieval

**Decision**: The first archive miner pass produces a deterministic normalized registry from archived OpenSpec artifacts:
- `scenario_seeds.json`
- `repair_patterns.json`
- `dtu_edge_cases.json`
- `exemplars.json`

Only after the deterministic layer is solid should more advanced retrieval or embedding-based ranking be added.

**Rationale**: Deterministic extraction is easier to validate, easier to test, and safer for dogfooding.

### D7: Dogfooding focuses on boundary-heavy coordinator scenarios first

**Decision**: This repository dogfoods the feature on the most software-factory-relevant surfaces:
- cross-interface coordinator scenarios
- multi-agent contention
- GitHub PR/check/review flows
- auth/transport degradation

**Rationale**: Those are exactly the kinds of behaviors that benefit from public/holdout separation, DTUs, and structured rework.

## Dogfood Rollout

### Stage 1: Scenario Visibility

- Classify current `agent-coordinator` gen-eval scenarios into public and holdout
- Preserve current scenario content, add manifest metadata first

### Stage 2: DTU-Lite Fixtures

- Add GitHub PR/check/review fixture pack
- Add transport degradation and auth degradation fixture packs

### Stage 3: Rework And Archive Intelligence

- Route dogfood validation failures into `rework-report.json`
- Generate the first `process-analysis` artifacts
- Mine archived changes and feed exemplars back into `/explore-feature`

## Resolved Questions

- **Holdout enforcement strategy**: Directory-based organization (`public/` and `holdout/` directories) reinforced by manifest metadata and skill-level filtering. No exclusion policies needed — clear path conventions are sufficient. (Resolved: D2)
- **Process-analysis generation**: Generated in `/validate-feature` only. Cleanup and merge gates consume it read-only. (Resolved: D5)
- **DTU probe depth**: First pass ingests public docs and examples only. Live probing is optional and enhances fidelity score but is never required. Low-fidelity DTUs remain eligible for public scenarios only. (Resolved: D3)
