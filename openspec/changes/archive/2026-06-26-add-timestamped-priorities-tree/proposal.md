# Add Timestamped Priorities Tree

## Why

`/prioritize-proposals` currently writes to a single file at `openspec/changes/prioritized-proposals.md` (and the matching `.json` when `--format json` is passed). Two problems compound:

1. **Wrong location.** The file lives under `openspec/changes/` but is not a change — it is a meta-report *about* changes. `openspec list`, `openspec validate`, and the change-archive workflow all treat `openspec/changes/` as the directory of in-flight proposals. The prioritization report is structurally a sibling of `docs/feature-discovery/opportunities.json`, not a sibling of `openspec/changes/add-foo/`.

2. **Each run overwrites the previous one, so the evolution of priorities is invisible.** Single-overwriting-file is fine when the only consumer is "what do we work on right now?", but it has three concrete failure modes against the broader goal of *tracking how priorities evolved*:
   - **Ad-hoc reruns that never get committed disappear.** An operator can re-prioritize three times in a session while exploring options; only the final state survives, and only if it gets committed.
   - **Squash-merges collapse intermediate runs.** When a PR that touched the report file is squash-merged, the intermediate states in the PR's commit history are lost from `main`.
   - **"What was the priority on date X?" via `git log` is much harder than `ls openspec/priorities/2026-05-*`.** The git-plumbing answer requires correlating commit dates against file mtimes; the filesystem answer is a single shell glob.

Empirical evidence the current pattern lets staleness hide: `openspec/changes/prioritized-proposals.md` is dated **2026-05-04**, ~5 weeks stale at the time of this proposal, with no warning surface — a reader has to open the file to discover the staleness.

The fresh pain trigger: today's `/explore-feature` run surfaced that ad-hoc reruns produce snapshots an operator wants to compare side-by-side, and the only way to do that with the current design is `git show <ref>:openspec/changes/prioritized-proposals.md`. A dated-directory layout makes side-by-side comparison a `diff openspec/priorities/<date-A>/report.md openspec/priorities/<date-B>/report.md`.

## What Changes

1. **New directory** `openspec/priorities/` containing per-run dated artifact directories.
   - **Path convention**: `openspec/priorities/<YYYY-MM-DD>-HHMMSS-<short-git-sha>/` (e.g., `openspec/priorities/2026-06-10-143052-a93fe59/`).
   - **Per-run files**: `report.md` (human-readable) and `report.json` (machine-readable, written only when `--format json` is passed — matches current skill behavior).
   - **Run-id format**: `HHMMSS-<short-git-sha>` — sortable within a day, ties each run to the commit it analyzed (useful for trend reproducibility), and human-recognizable.

2. **Stable "latest" pointer** as **rewritten flat files** (not symlinks).
   - `openspec/priorities/latest.md` and `openspec/priorities/latest.json` are rewritten on every run with the same content as the freshly-written dated artifacts.
   - Rationale: cross-platform compatibility (Windows, archive tarballs, GitHub web-UI downloads), clean `git diff` showing what changed since last run, no symlink-resolution gotchas in CI.

3. **Mandatory artifact header on `report.json`** matching the codeviz event-artifact schema (codeviz roadmap, Phase 0):
   ```json
   {
     "schema_version": 1,
     "generated_at": "2026-06-10T14:30:52Z",
     "git_sha": "a93fe59",
     "generator": "prioritize-proposals@1.0",
     "run_id": "2026-06-10-143052-a93fe59",
     "event_kind": "priorities-report"
   }
   ```
   Inline implementation (~10 LOC) inside `skills/prioritize-proposals/scripts/`. **Does not block on** the codeviz roadmap's `skills/shared/artifact_header.py` helper; when that helper lands, this skill migrates to it via a small follow-up.

4. **Retention policy** — keep ~30 most recent dated directories by default.
   - Configurable via a new `--retain N` flag on `/prioritize-proposals`.
   - Trigger: every run, after writing the new artifacts, scan `openspec/priorities/` and act on entries older than the Nth most recent.
   - **What happens to dropped entries — decision deferred to Gate 1** (see Approaches Considered → Decision A below). Both options can be implemented; this proposal asks the operator to pick the default behavior.

5. **Update `skills/prioritize-proposals/SKILL.md`** to write to the new location. Update `openspec/specs/skill-workflow/spec.md` to reflect the new output paths. Update the canonical skill only — `.agents/skills/` and `.claude/skills/` mirrors are auto-regenerated by `skills/install.sh`.

6. **Migration of the existing stale `openspec/changes/prioritized-proposals.md`** — **decision deferred to Gate 1** (see Approaches Considered → Decision B). Both paths are clean; this proposal asks the operator to pick.

## What Doesn't Change

- The content of the priorities report itself (scoring formula, sections, JSON shape *except* for the new header block).
- The `--format md|json` flag semantics — markdown is always written; JSON is written only on `--format json`, same as today. The `latest.json` only exists if at least one run has used `--format json`.
- The relationship between `/prioritize-proposals` and its consumers — no other skill or script reads the report file programmatically (verified by grep). Only the skill itself, the `openspec/specs/skill-workflow/spec.md` documentation, and an archived 2026-02-08 change reference the path.
- `docs/feature-discovery/opportunities.json` and `docs/feature-discovery/history.json` — these are sibling artifacts from `/explore-feature`, not affected by this change.
- `.gitignore` — the dated directories are tracked in git (small text files, low repo bloat, audit-trail value).

## Approaches Considered

### Approach 1: Dated directories with flat-file `latest` (Recommended)

**Description**: Per-run artifacts in `openspec/priorities/<YYYY-MM-DD>-HHMMSS-<short-git-sha>/`, rewritten flat-file `latest.md` + `latest.json` at the top level. Retention via a `--retain N` flag (default 30).

**Pros**:
- Mirrors the existing `openspec/changes/archive/<YYYY-MM-DD>-<id>/` convention — operators already understand "dated directory per artifact."
- Multi-file per run is natural (md + json + future `findings.json` or `delta-from-previous.json` follow-ons).
- Trend analysis is a one-liner: `jq -s '.' openspec/priorities/*/report.json`.
- `latest.{md,json}` gives cheap "most recent" access without globbing.
- Cross-platform: no symlinks; works on Windows, in tarballs, in GitHub web-UI downloads.
- Staleness surfaces at the directory-name level (`ls openspec/priorities/` shows the most recent date), instead of hiding inside a stale file.
- Aligns with the codeviz event-artifact convention (dated path, mandatory header, accumulating not overwriting).

**Cons**:
- Repo grows by ~5 KB per run × ~30 runs in-tree = ~150 KB sustained; bounded by retention.
- Two new flat files (`latest.md`, `latest.json`) duplicate the most-recent dated artifact's content — diff-friendly but not single-source-of-truth.
- Two follow-up decisions deferred to Gate 1 (retention behavior, migration of stale file).

**Effort**: **S** (~250 LOC net: skill bash updates + small Python helper for path construction + retention scan + flat-file writer + spec update + migration step).

### Approach 2: Single append-only JSONL log

**Description**: Each run appends one JSON line to `openspec/priorities/history.jsonl`. No per-run directories. The "latest" view is `tail -1 openspec/priorities/history.jsonl | jq`. A separate `latest.md` is rewritten each run from the most recent line.

**Pros**:
- Single file → trivial append-only semantics; no path-construction logic.
- Trend analysis is a `cat history.jsonl | jq` instead of `jq -s '.'`.
- No retention pruning needed — the file is just-text; size grows linearly but slowly.

**Cons**:
- **report.md doesn't fit.** Markdown isn't lineable; you'd have to either drop the markdown view entirely (regression in human readability) or write markdown to a separate dated file anyway (defeating the "single file" advantage).
- Side-by-side comparison of two specific runs requires `jq` extraction, not a directory `diff`.
- File grows unbounded without an explicit retention step (which then needs a JSONL-aware truncator, more complex than `rm -rf <dir>`).
- Mismatches the existing OpenSpec archive idiom — operators have no precedent for "append-only log" in this repo.

**Effort**: **S** (~150 LOC, but with the regression noted above).

### Approach 3: Status quo + git-only history

**Description**: Leave `openspec/changes/prioritized-proposals.md` in place. When operators want to see priority evolution, they use `git log --follow openspec/changes/prioritized-proposals.md` and `git show <ref>:openspec/changes/prioritized-proposals.md`.

**Pros**:
- Zero new code.
- Git is already authoritative for committed state.

**Cons**:
- **Does not address any of the three named failure modes** (ad-hoc reruns lost, squash-merges collapsing, queryability friction).
- File remains semantically misplaced under `openspec/changes/`.
- Staleness still hides inside the file rather than surfacing at the directory level.
- "What was the priority on date X?" remains a multi-step git-plumbing operation.

**Effort**: **XS** (zero).

### Selected Approach

**Approach 1: Dated directories with flat-file `latest`** — selected at Gate 1 (Direction Approval).

Gate 1 decisions baked in:
- **Decision A → A2**: Runs past the retention threshold are **archived in repo** at `openspec/priorities/archive/<YYYY-MM-DD>-HHMMSS-<short-git-sha>/`, kept in git permanently. Accepts ~2 MB/year sustained growth in exchange for full audit trail.
- **Decision B → B2**: The existing stale `openspec/changes/prioritized-proposals.md` is **moved as a legacy entry** to `openspec/priorities/2026-05-04-legacy/report.md`, becoming the first historical entry in the new tree. Requires a small missing-header allowance for pre-migration entries.

### Recommended: Approach 1

Approach 1 wins because:
1. **It addresses the named failure modes.** Approaches 2 and 3 either degrade an existing capability (markdown view) or leave the original problem unsolved.
2. **It composes with the codeviz event-artifact convention.** When the `skills/shared/artifact_header.py` helper lands (codeviz Phase 0), this skill gets it for free.
3. **It matches the existing OpenSpec archive idiom.** Operators don't need a new mental model — `openspec/priorities/<date>-<id>/report.md` reads exactly like `openspec/changes/archive/<date>-<id>/proposal.md`.
4. **The cons are bounded.** Repo bloat is capped by retention; the two deferred Gate 1 decisions are small enough that picking either way is acceptable.

### Gate 1 Decisions

This proposal asks the operator to pick two small defaults at Direction Approval:

**Decision A: What happens to runs past the retention threshold?**
- **A1 — Delete them.** Older runs are permanently removed. Pros: zero repo bloat over time. Cons: irrecoverable; an operator who wants to see "the priority during the Q1 push" can't get it back.
- **A2 — Move to `openspec/priorities/archive/<YYYY-MM-DD>-HHMMSS-<short-git-sha>/`.** Older runs accumulate under an archive subdir, kept in git. Pros: full audit trail. Cons: unbounded growth (mitigated by being small text files; ~5 KB × 365 = ~2 MB/year sustained).
- **A3 — Hybrid: archive on rolling 90-day window, delete entries older than that.** Pros: bounded growth, recent history preserved. Cons: most complex retention logic; 90-day boundary is arbitrary.

**Decision B: What happens to the existing stale `openspec/changes/prioritized-proposals.md`?**
- **B1 — Delete it.** It is 5 weeks stale and not trusted. Pros: clean cut. Cons: loses the historical entry from May 2026.
- **B2 — Move-and-rename to `openspec/priorities/2026-05-04-legacy/report.md` as the first historical entry.** Pros: preserves the audit trail; the legacy entry slots cleanly into the new tree. Cons: the legacy file lacks the new mandatory header (would need a "legacy" event_kind variant or a missing-header allowance for pre-migration entries).

## Out of Scope

- A web UI or CLI for browsing the priorities history. The directory + glob substrate is enough for `jq` / `diff` / `ls` workflows; a richer browser is a follow-up if pain emerges.
- Cross-repo priorities aggregation. This change is single-repo scope.
- Migrating `docs/feature-discovery/opportunities.json` to the same dated-tree pattern. They share the spirit but `opportunities.json` already has a sibling `history.json` index that solves the evolution-tracking problem; no fresh pain there.
- Adopting the codeviz `skills/shared/artifact_header.py` helper. That helper does not exist yet (roadmap-planned). This change inlines the ~10-LOC header construction and migrates to the helper when it lands.
- A CI lint enforcing the mandatory header on `report.json`. The schema is documented; a CI check is a codeviz roadmap concern.
- Removing the `--format` flag or changing its semantics. The flag's behavior is preserved exactly.
