### Decisions
1. **Local web app (Vite+React+TS) as v1 frontend, not Mac-native or TUI** `architectural: coordinator-kanban-viz` — Works in cloud-harness sessions (a primary user), local Mac/Linux, and Tauri-wrapped native from one codebase. Composes with codeviz's eventual SPA on the same toolchain.
2. **SSE (one-way) for live updates, not WebSocket** `architectural: coordinator-kanban-viz` — Live data flows server→client only. SSE reuses HTTP auth, traverses proxies, has trivial client surface (EventSource). WebSocket framing/ping-pong adds no UX value on a read-mostly board. 3. **No JSON sidecar mirror of work_queue state** `architectural: coordinator-kanban-viz` — Coordinator remains the single source of truth. A mirrored docs/kanban-viz/state.json would duplicate the source and recreate the display-drift problem that add-coordinator-task-status-renderer was built to solve. Only saved views (git-committed, user-curated) and audit events (event-class, dated paths) persist to disk.
4. **FalkorDB work-state schema reserved but NOT implemented in v1** `architectural: coordinator-kanban-viz` — Codeviz Phase 0 introduces FalkorDB; v1 of the Kanban predates it. To avoid a future rename pass we reserve node labels (WorkPackage, Agent, Vendor, Worktree, Lock, SyncPoint, AuditEvent) and edge types (CLAIMED_BY, BLOCKED_ON, LOCKS_FILE, WORKING_IN, RAN_BY, BLOCKS_SYNCPOINT, IMPLEMENTS_TASK) in docs/kanban-viz/falkordb-reservation.md. Codeviz Phase 0 ingestion adopts these without coordination overhead.
5. **Codeviz operation reversibility taxonomy applied to every UI write action** `architectural: coordinator-kanban-viz` — read auto-allowed; reversible-write (save-view, drag-to-Ready) auto-allowed + audit; destructive-write (force-release-lock, kick-agent) requires per-action consent prompt + audit on both confirm and decline. Reversibility classifier hard-coded in v1 with a comment pointing to the codeviz reservation in skills/shared/op_reversibility.py for future convergence.
6. **Tauri scaffold ships in v1; production Tauri build does NOT** `architectural: coordinator-kanban-viz` — cargo check verifies the scaffold compiles; CI does not build/sign/notarize a binary. Lowers follow-up cost for native distribution to ~100 lines without committing to App Store distribution prematurely. Browser code paths run without @tauri-apps/api evaluated; Tauri APIs feature-detected at runtime.
7. **Saved views and audit events follow codeviz storage tier policy** `architectural: coordinator-kanban-viz` — Saved views → docs/kanban-viz/saved-views/<slug>.json (git, mandatory artifact header). Audit events → docs/kanban-viz/audit/<YYYY-MM-DD>/<run-id>.json (event-class, dated, mandatory header, 90-day in-tree retention per codeviz policy). No new substrate.

### Alternatives Considered
- Approach 2: Mac-native Swift app (Storybloq style): rejected because Cannot run in cloud-harness sessions, which are a primary use case. Single-platform; adds Swift toolchain to a Python+TS codebase; App Store distribution incompatible with the repo's local-first ethos.
- Approach 3: TUI-only (Python Textual terminal Kanban): rejected because Cannot render vendor swimlanes legibly at typical terminal widths; no drag-to-approve interaction reduces UI to read-only and loses one of the MVP's differentiators; forces an eventual GUI rebuild — duplicate maintenance.
- WebSocket transport for live updates: rejected because Bidirectional framing/ping-pong/reconnection complexity for no UX benefit on a read-mostly board. SSE is one-way, HTTP/1.1-native, proxy-friendly, and has a trivial client surface (EventSource).
- JSON sidecar mirroring work_queue state to disk: rejected because Duplicates the source of truth and recreates the display-drift problem that add-coordinator-task-status-renderer exists to solve. Violates codeviz storage tier policy (mutable queryable state lives in Postgres, not git).
- Pure read-only observer (defer all write actions): rejected because Considered at Gate 1 discovery (v1 scope question). User selected control surface explicitly: the most-asked operator question ("why can't I merge") is best answered by surfacing the blocker + offering a consent-gated kick — not just labeling it.
- Dedicated GET /issues/by-change/<id> convenience endpoint on coordinator: rejected because Existing GET /issues?labels=change:<id> is sufficient at OpenSpec scale (10-30 tasks/change). Convenience endpoint saves one label-filter step but adds API surface. Same call the add-coordinator-task-status-renderer change reuses.

### Trade-offs
- Accepted Node toolchain in a Python-dominant repo over single-language stack (Python TUI only) because Containment via apps/kanban-viz/ scope + .nvmrc pin + no cross-cutting impact. Buys browser + Tauri-native + (eventually) codeviz SPA from one codebase.
- Accepted Per-commit Tauri scaffold ships without production build pipeline over full Tauri build wired into CI in v1 because Defers signing/notarization/distribution to a follow-up while making the eventual native wrap ~100 lines instead of a re-architecture. cargo check guards correctness.
- Accepted Hard-coded reversibility table in frontend TypeScript over single shared classifier from skills/shared/op_reversibility.py because Codeviz reserves the shared classifier; v1 of this Kanban predates it. Intentional duplication with a comment pointing to the convergence target. Cleanup at Phase 0.
- Accepted Single-repo scope in v1 over cross-repo board (coordinator is already multi-repo capable) because Cross-repo tab strip is a follow-up after the single-repo UX is validated. Reduces v1 surface area while keeping the data path multi-repo-capable on the server side.
