# Design: Live Service Testing Pipeline

**Change ID**: `live-service-testing`

## Architecture

```
┌─────────────────────────────────────────────────────┐
│                  Validation Pipeline                 │
│                                                     │
│  /implement-feature ──► phase_deploy + phase_smoke  │
│       (soft gate)        │                          │
│                          ▼                          │
│  /cleanup-feature ───► check validation-report.md   │
│       (hard gate)    ──► re-run if missing/failed   │
│                                                     │
├─────────────────────────────────────────────────────┤
│                  Phase Runner Layer                  │
│                                                     │
│  phase_deploy.py ──► TestEnvironment.start()        │
│                  ──► TestEnvironment.wait_ready()    │
│                  ──► persist .test-env               │
│                                                     │
│  phase_smoke.py ──► load .test-env                  │
│                 ──► pytest smoke_tests/              │
│                 ──► update validation-report.md      │
├─────────────────────────────────────────────────────┤
│              TestEnvironment Protocol                │
│                                                     │
│  ┌──────────────────┐  ┌────────────────────────┐   │
│  │ DockerStackEnv   │  │ NeonBranchEnv          │   │
│  │                  │  │                        │   │
│  │ port_allocator   │  │ neonctl / Neon API     │   │
│  │ docker compose   │  │ pg_dump / migrations   │   │
│  │ pg_isready       │  │ branch lifecycle       │   │
│  └──────────────────┘  └────────────────────────┘   │
│                                                     │
├─────────────────────────────────────────────────────┤
│              Smoke Test Suite                        │
│                                                     │
│  test_health.py ── test_auth.py ── test_cors.py     │
│  test_error_sanitization.py                         │
│                                                     │
│  Parametrized by: API_BASE_URL, POSTGRES_DSN        │
└─────────────────────────────────────────────────────┘
```

## Design Decisions

### D1: TestEnvironment as Protocol, not ABC

Use `typing.Protocol` rather than an abstract base class. This allows duck-typing — any object with `start()`, `wait_ready()`, `teardown()`, `env_vars()` methods qualifies. The phase runners don't need to import the concrete implementations.

**Why**: Consistent with the project's composition-over-inheritance convention (per `openspec/project.md`). Also avoids import-time dependency on `neonctl` when using Docker and vice versa.

### D2: .test-env file as phase handoff

Phase deploy writes a `.test-env` file (dotenv format) that phase smoke reads. This decouples the phases — they can be run in the same process or as separate CLI invocations.

**Why**: The validate-feature SKILL.md already expects phases to be independently invocable. Persisting to a file matches the existing pattern of `validation-report.md` as inter-phase state.

**Format**:
```bash
# .test-env — generated by phase_deploy.py
TEST_ENV_TYPE=docker  # or "neon"
POSTGRES_DSN=postgresql://postgres:postgres@localhost:10000/postgres
API_BASE_URL=http://localhost:10003
COMPOSE_PROJECT_NAME=ac-a1b2c3d4
NEON_BRANCH_ID=br-abc123  # only for neon
SEED_STRATEGY=migrations  # dump_restore or migrations
STARTED_AT=2026-03-31T20:00:00Z
```

### D3: Port allocator — direct import over MCP call

Import `port_allocator` from `agent-coordinator/src/` directly rather than going through the MCP `allocate_ports` tool. The stack launcher runs in the same Python process and doesn't need RPC overhead.

**Why**: The MCP tool is a thin wrapper around the same `PortAllocator` singleton. Direct import is simpler and avoids requiring a running coordinator just to allocate ports for local testing.

**Fallback**: If the import fails (e.g., running from a different venv), fall back to MCP tool call if coordinator is available.

### D4: Neon branching via neonctl CLI over API SDK

Use `neonctl branches create` subprocess call rather than the Neon Python SDK.

**Why**:
1. The `neon-postgres` skill already documents `neonctl` usage patterns
2. Avoids adding `neon-api` as a Python dependency to the skills venv
3. CLI output is parseable JSON (`--output json`)
4. Works from any environment where neonctl is installed (CI, local dev)

### D5: pg_dump for dump_restore strategy

Use `pg_dump --format=custom` from local ParadeDB, `pg_restore` into Neon branch. Custom format handles cross-version compatibility and supports parallel restore.

**Why**: Custom format is recommended for production use. Plain SQL format would break on ParadeDB-specific extensions that Neon doesn't support. Custom format with `--no-owner --no-privileges` strips ownership info that doesn't apply to Neon.

**Extension handling**: Use `--exclude-extension=pg_search` in pg_dump to skip ParadeDB-only extensions. pgvector is supported on both platforms.

### D6: Seed data as SQL, not Python fixtures

The seed data lives in `agent-coordinator/supabase/seed.sql` as plain SQL, not as Python fixture factories.

**Why**:
1. Reusable across both Docker (via docker-entrypoint-initdb.d) and Neon (via psql pipe)
2. The seed.sql file already exists (currently empty) — this is the expected location
3. Deterministic and auditable — diffs in git show exactly what test data changed
4. No Python dependency for seeding — works with just psql

### D7: Soft/hard gate via validation-report.md

Both gates check for a `## Smoke Tests` section in `validation-report.md` with a `status: pass` line. The soft gate in `/implement-feature` creates the section with `status: skipped` if Docker/Neon unavailable. The hard gate in `/cleanup-feature` rejects `skipped` or `fail`.

**Why**: Reuses the existing `validation-report.md` artifact pattern from `validate-feature/SKILL.md`. The report is already the canonical record of what validation has passed.
